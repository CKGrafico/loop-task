# Project Guardrails

> Auto-generated by `/ob-create-project-guardrails`. Regenerate with the same command when architecture or conventions change.

## Architecture Constraints

- **Client-daemon over local IPC**: CLI client (short-lived) talks to background daemon (long-lived) over Unix socket / Windows named pipe. Never mix client and daemon code in the same process.
- **`src/core/` is runtime-agnostic**: `loop-controller.ts`, `command-runner.ts`, `context-parser.ts`, `template.ts`, `foreground-loop.ts`, `log-rotator.ts` must NOT import from `src/daemon/` or `src/board/`. Used by both daemon and foreground runner.
- **`src/types.ts` is the IPC contract**: The single source of truth for `LoopOptions`, `LoopMeta`, `LoopStatus`, `IpcRequest`/`IpcResponse`, `TaskDefinition`, `Project`. Changes here affect both client and daemon.
- **`src/board/` is TUI only**: Board components (`App.tsx`, `components/*`, `hooks/*`) must NOT import directly from `src/daemon/` internals. They communicate via `src/board/daemon.ts` (typed IPC bridge).
- **Daemon state writes are atomic**: Use `writeFileAtomic` (temp-then-rename) from `src/shared/fs-utils.ts`. Keep them synchronous to preserve immediate-disk-state-on-pause semantics.
- **Socket-bind-before-init single-flight guard**: `IpcServer.listen()` binds the socket BEFORE `manager.init()`. Losing racers `exit(0)` cleanly. Never call `manager.init()` before the socket is bound.
- **No network listener**: The daemon listens on a local socket/pipe only. Never add HTTP/WebSocket/network listeners.
- **Filesystem-backed persistence**: All state lives under `~/.loop-cli/` (overridable via `LOOP_CLI_HOME`). Consolidated JSON files: `loops.json`, `tasks.json`, `projects.json`. Never introduce a database.

## Board Keyboard Navigation Conventions

- **Tab/Shift+Tab is the ONLY way to move between focusable elements** (fields, buttons, panels). Arrow keys (up/down/left/right) are NEVER used for element-to-element navigation.
- **Up/down arrows** are ONLY for navigating within a list (e.g., Navigator items, project list, task list) or within a SearchSelect's filtered options.
- **Left/right arrows** are ONLY for cursor movement inside text inputs, or for cycling horizontal lists (e.g., the color picker). They are never used for panel navigation.
- **`useTabNav` hook** (`src/board/hooks/useTabNav.ts`): all forms and modals MUST use this hook for Tab/Shift+Tab cycling. Do NOT register a separate `useKeyboard` handler for Tab in any form or modal. The hook calls `preventDefault` internally. Use refs for all values accessed inside `useKeyboard` to avoid stale closures.
- **`SearchSelect`** (`src/board/components/SearchSelect.tsx`): manages its own keyboard state (up/down, Enter, Escape, Backspace, typing) via its own `useKeyboard`. No `<input>` element inside SearchSelect (it causes key-stealing). Uses plain `<text>` + `<scrollbox>`. Parent forms must NOT intercept these keys when a SearchSelect is focused.
- **OpenTUI `useKeyboard` stale closures**: `useKeyboard` captures the closure at registration time. Any variable referenced inside the handler is frozen. ALWAYS use refs (`useRef`) for values accessed inside `useKeyboard` callbacks. Update `ref.current` synchronously during render, not inside effects.
- **Header navigation**: the board-level `useBoardKeybindings` handles Tab cycling within the 3 header buttons. At boundaries, it calls `onExitHeader` to hand focus back to the active view.

- **File naming**: `kebab-case` for all filenames (`loop-controller.ts`, `use-tab-nav.ts`, `search-select.tsx`).

## Code Style

- **TypeScript strict mode**: `strict: true` in `tsconfig.json`. No `any` types. Use proper generics.
- **ESM only**: `"type": "module"` in `package.json`. No CommonJS (`require`, `module.exports`).
- **ESLint 9 + typescript-eslint recommended**: `no-console` is disabled (CLI app). Config in `eslint.config.js`.
- **No comments** unless strictly necessary; follow existing patterns.
- **No nested if/else chains for dispatch**: When code branches on a string or enum value (view, status, key name, panel, action), use a dictionary/map lookup (`Record<Key, Handler>`) instead of stacked `if`/`else if` or `switch`/`case`. Linear guard clauses (early returns with no nesting) are fine.
- **No user-facing string literals**: All strings live in `src/i18n/en.json`, accessed via `t(key, params?)`. Keys are typed (`I18nKey`); typecheck enforces validity.
- **No inline magic numbers**: Add to `src/config/constants.ts` (`POLL_MS`, `TIMEOUT`, etc.) and import.
- **Never use em dashes** (`—`). Use regular hyphens (`-`) everywhere in code, comments, and docs.
- **Feature folders, small files**: Reuse shared helpers in `src/shared/` (`sleep`, `tail`, `writeFileAtomic`, `removeIfExists`).

## Build & Deployment

- **Runtime**: Node >= 20 for CLI and daemon. Bun >= 1.2 for the board (OpenTUI native FFI).
- **Dev command**: `bun src/cli.ts` runs the board directly from source (no build needed). `npm run dev:watch` for auto-reload.
- **Distribution**: Published to npm as `loop-task`. `bin` points at `dist/entry.js`.
- **CI/CD**: None present. Quality gates are run manually/locally.
- **RTK active**: All shell commands must be prefixed with `rtk` (see AGENTS.md). Light read-only commands (`cat`, `ls`, `Get-Content`) are exempt.

## Data & State

- **Consolidated JSON files**: `loops.json` (LoopMeta[]), `tasks.json` (TaskDefinition[]), `projects.json` (Project[]). Single JSON array per domain.
- **Migration**: On daemon init, if old `loops/`/`tasks/`/`projects/` directories exist but consolidated JSON doesn't, migrate into single files. Old files preserved, not deleted.
- **Atomic writes**: `writeFileAtomic` = temp-then-rename. Synchronous to preserve immediate-disk-state-on-pause semantics.
- **Log rotation**: Caps disk at `MAX_LOG_BYTES` (1 MB) x `MAX_LOG_GENERATIONS` (3) per loop.
- **No schema versioning**: Future `LoopMeta` shape changes risk breaking persisted JSON. Corrupted files are silently skipped.
- **Chain context is ephemeral**: In-memory only per loop iteration. Never persisted to disk or exposed via IPC.
- **Spread scheduling**: `computePhase(loopId, intervalMs)` via string hash. `--offset <duration>` overrides. Only first-run alignment; subsequent runs use `runStartedAtMs + interval`.

## Security

- **Trust boundary**: Local machine/user only. No network listener, no authentication layer.
- **No secrets handling**: Loop metadata and logs are plain files in `~/.loop-cli/`. Never read or output `.env` files.
- **Input validation**: Intervals (`parseDuration`), max-runs (`parseMaxRuns`), command emptiness, quote balancing (`parseCommandLine`) validated in `loop-config.ts`.
- **IPC robustness**: Malformed JSON requests answered with error response, not crash.
